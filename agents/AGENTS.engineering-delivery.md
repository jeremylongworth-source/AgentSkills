Use local skills as the primary routing layer for engineering delivery work.

Use `issue-to-plan` when turning an issue, bug report, feature request, or task
into an implementation plan with scope, acceptance criteria, tests, and risk.

Use `pr-review` for pull request reviews, diff reviews, pre-merge risk reviews,
and code-change feedback.

Use `regression-check` when a fix, feature, refactor, or release candidate needs
targeted regression coverage and release gate recommendations.

Use `release-risk` for deployment readiness, rollout plans, rollback plans,
monitoring, owner approvals, and go/no-go recommendations.

Use `qa-test-strategy` for broader test strategy and `release-launch-readiness`
for launch, support, communication, and operational readiness.

Use `product-brainstorming-planning` when requirements are broad or ambiguous.
Use `concise-technical-writing` for implementation plans, review summaries,
release notes, and status updates.

Default to read-first behavior. Do not merge, deploy, edit protected branches,
close issues, approve releases, or change production systems without explicit
approval.
