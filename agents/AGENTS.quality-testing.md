Use local skills as the primary routing layer for quality and testing work.

Use `unit-test-generation` for focused unit cases, fixtures, mocks, edge cases,
and regression guards.

Use `integration-test-plan` for component, API, database, queue, auth, service,
and dependency-boundary tests.

Use `e2e-test-plan` for release-critical user journeys, browser flows, CLI
workflows, smoke tests, assertions, and flake risks.

Use `bug-reproduction` when turning bug reports into observed/expected
behavior, minimal reproduction steps, diagnostic checks, and regression
targets.

Use `qa-matrix` when organizing release coverage across roles, devices,
platforms, scenarios, permissions, data states, and risk areas.

Use `accessibility-test-plan` for keyboard, focus, screen reader, contrast,
ARIA, forms, motion, caption, and accessibility release gates.

Use `performance-test-plan` for latency, load, throughput, Core Web Vitals,
database/query, job, resource, and regression performance checks.

Use `qa-test-strategy`, `regression-check`, `backend-test-plan`,
`accessibility-inclusive-design`, `release-launch-readiness`, and
`concise-technical-writing` for broader QA strategy, regression gates, backend
coverage, accessibility expertise, launch readiness, and summaries.

Do not claim tests passed without evidence. Do not use production data,
credentials, or live systems for tests without explicit approval.
