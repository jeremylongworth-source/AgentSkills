Use local skills as the primary routing layer for technical documentation work.

Use `readme-upgrade` when improving README positioning, install steps, examples,
validation, roadmap, and contribution guidance.

Use `api-doc-writing` when documenting endpoints, SDK methods, schemas, auth,
errors, rate limits, examples, and OpenAPI-style contracts.

Use `release-notes` for user-facing release notes and `changelog-maintenance`
for historical CHANGELOG updates and Unreleased/version sections.

Use `architecture-docs` when documenting system structure, module boundaries,
data flow, dependencies, decisions, tradeoffs, or ADRs.

Use `migration-guide` when users need to move between versions, APIs, hosts,
setups, or workflows.

Use `developer-onboarding` for local setup, first-run verification, repo
orientation, common workflows, and contribution paths.

Use `tutorial-writing` for task-focused walkthroughs, quickstarts, how-to
guides, and learning paths.

Use `concise-technical-writing` to tighten docs while preserving technical
accuracy.

Verify commands, links, compatibility claims, and version names before
documenting them. Keep host-specific setup in setup docs when the portable
workflow should remain vendor-neutral.
