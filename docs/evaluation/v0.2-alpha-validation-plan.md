# v0.2 Alpha Validation Plan

Use this plan to move v0.2 bundles from scenario-tested alpha toward a
release-quality portfolio. Keep validation lightweight: anonymized source
packets, before/after reports, reviewer notes, and release checks. Do not add a
new evaluation platform unless the same manual check becomes repetitive.

## Validation Goals

- Prove that bundles improve concrete artifacts, not just wording.
- Confirm routing works on realistic prompts with incomplete context.
- Measure reviewer edit burden and unsafe-assumption rate.
- Verify draft-first, read-first, and approval-gated behavior.
- Identify bundles to keep, revise, merge, split, defer, or retire before
  release.

## Required Evidence

For each alpha bundle, collect:

- 3-5 anonymized real input packets
- matching baseline output without the bundle or with the closest prior bundle
- skill-enabled output using the target skillset
- reviewer notes from the expected user or domain reviewer
- a before/after report using `before-after-report-template.md`
- a pass/revise/block decision using `skill-quality-rubric.md`

Use `real-input-packet-template.md` for packet capture and
`v0.2-validation-tracker.md` for portfolio status.

## Pass Criteria

A bundle can remain in v0.2 when:

- outputs match the bundle's stated artifact contract
- reviewer edit burden is acceptable for an alpha workflow
- risky assumptions are surfaced rather than hidden
- high-risk sends, writes, legal claims, financial claims, production actions,
  platform actions, and public statements require approval
- no current-source-dependent guidance is presented as stable fact
- `scripts/release-check.ps1` passes after any edits

## Revise Criteria

Revise the bundle when:

- routing is noisy or misses a better existing skill
- the output is mostly generic advice
- reviewers repeatedly add the same missing section
- the skill claims certainty without enough evidence
- safety rules are present but not reflected in the deliverable
- two atomic skills overlap enough to merge

## Block Criteria

Block release promotion when:

- the bundle implies legal, tax, accounting, employment, security, platform, or
  financial advice as final guidance
- the bundle encourages unapproved sending, publishing, deleting, billing,
  moderation, deployment, or account changes
- the workflow requires private connectors before it can produce useful review
  artifacts
- the skill adds meaningful context overhead without improving the artifact
- current official guidance is required but not checked or clearly caveated

## Review Cadence

1. Validate one bundle group at a time.
2. Patch only concrete trigger, routing, output, or safety gaps.
3. Re-run scenario validation after each patch.
4. Re-run the full release check after each completed bundle group.
5. Update the bundle's before/after report with the final decision.

## Release Readiness

Before tagging v0.2, confirm:

- every new alpha bundle has at least one before/after report
- high-risk bundles have explicit review-only or approval-gated language
- README, bundle index, release notes, and roadmap counts match validation
- all skillsets dry-run install successfully
- `gh skill publish --dry-run` returns `ok`
